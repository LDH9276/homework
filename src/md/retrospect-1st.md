# 첫 번째 회고

첫 번째 부트캠프는 내가 기존에 배웠던 HTML 같은 기본 문법이 아닌, **CLI(Command Line Interface)**와 **Git**부터 바로 시작해서 꽤 당황스러웠다.  
하지만 그만큼 시간이 어떻게 가는지 모를 정도로 빠르게 흘렀고, 많은 것을 배우고 경험할 수 있었던 꽉 찬 한 주였다.

물론 나는 기존에 **웹 퍼블리셔 경험**이 있어 HTML/CSS/JavaScript는 어느 정도 알고 있었다.  
하지만 익숙한 GUI 환경이 아닌, CLI 환경과 Git 명령어부터 시작하니 개발의 '진짜 기초'를 다지는 느낌이라 상당히 흥미로웠다.

## 1. 기초부터 탄탄하게, CLI와 친해지기

가장 먼저 했던 것은 CLI 환경과 Git 문법에 익숙해지는 것이었다.  
이는 개발의 가장 기초이기도 하고, **"개발자들은 게을러야 한다"**는 말처럼 익숙해지면 GUI보다 훨씬 편하기 때문이다.

다만, 나 역시 GUI 환경에 익숙하다 보니 처음에는 꽤 버벅거렸다.  
특히 rm(삭제)과 cp(복사) 같은 명령어들이 낯설어서 이 부분 위주로 집중해서 연습했다.

가령, `rm(삭제)`와 `cp(복사)`.  
그 때문에 이 부분을 중점으로 연습했던 것 같다.

## 2. Git: 헷갈리지만 익숙해져야 하는 도구

전의 회사에서 git을 쓰긴 했지만, 나는 `push, commit, pull`만 사용했다.  
여기서 가장 크게 배운건 checkout 대신 `switch, branch` 였던 것 같다.  
거의 습관처럼 썼지만 `switch, branch` 문법에 익숙해져야 했었다.

또한 `pull`만 썼는데 `fetch`로 해당 리포지토리나 브랜치의 변경 점을 추적하고
`difftool`로 각 두개의 차이를 확인 할 수 있었다.

항상 타성에 젖어 '왜? 이렇게 쓰는거지?'에 대한 답이라서
명확하게 쓰면서 느낄 수 있었다.

```
git pull origin/dev
# 1. 원격 변경 사항을 가져와서(Fetch)
# 2. 바로 내 코드와 합침(Merge)
# -> 충돌이 나면 그때 해결 (다소 위험할 수 있음)
```

라는 `pull` 방식에서 두 개의 충돌을 방지하기 위해
`fetch` 후 `merge`라는 방법은 처음으로 배운 것 같다.

```
git fetch dev                  # 1. 원격 변경 사항만 일단 가져오기 (병합 X)
git difftool dev origin/dev    # 2. 내 코드와 차이점을 눈으로 비교
git merge origin/dev           # 3. 충돌 수정 후 안전하다고 판단되면 병합
```

git pull이 내부적으로는 fetch와 merge를 합친 것이라는 사실을 알게 되니, 두 방식의 차이가 명확히 보였다.
안전한 협업을 위해 앞으로는 뒤에 배울 `PR, Pull Request`나 후자의 방식을 더 많이 연습하게 될 것 같다.
아쉬웠던 점은 `rebase`인데, 이 부분은 추후 학습하여 추가 회고에 남겨보려 한다.

## 3. git ignore와 npm

이전에는 연습 삼아 `npm`을 Vite나 Swiper 같은 라이브러리 설치 도구로만 사용했었다.  
`package.json` 파일이나 `.gitignore`는 중요하게 생각하지 않고 대충 넘기곤 했다.

하지만 이번 기회에 `npm`을 통해 `.gitignore`를 자동으로 생성해 주는 add-gitignore 기능과,  
`package.json`의 `scripts`를 활용해 복잡한 명령어를 단순화하는 법을 배웠다.  
패키지 관리부터 실행 자동화까지, 단 이틀간의 수업이었지만 개발자로서의 기본 소양과 수준을 한 단계 끌어올릴 수 있었던 귀중한 시간이었다.
